# 日志保留上限 10k 性能验证报告

日期：2026-06-22

## 结论

前端日志保留上限已从 1,000 条提高到 10,000 条。通过 WebSocket 批处理、搜索索引缓存和零拷贝时间排序，10k 场景下处理 500 条新增日志的中位耗时为 **7.36ms**。

相对于“只修改上限、不优化算法”的 10k 实现：

- 热路径耗时降低 **99.57%**
- 吞吐性能提高约 **231.6 倍**
- React 状态提交次数从 500 次降至 20 次，减少 **96%**

优化后的 10k 热路径也比原有 1k 实现快约 **24.7 倍**。

## 修改内容

1. 前端默认日志上限由 1,000 提高到 10,000。
2. WebSocket 日志先进入内存队列，每 50ms 批量提交一次 React 状态。
3. 批量合并使用单次定长数组分配，避免每条日志执行展开复制和 `slice()`。
4. 日志内部保持“新记录在前”的顺序；默认倒序直接读取原数组，正序通过索引换算读取，不再复制和排序 10k 数组。
5. 搜索使用 `useDeferredValue` 保持输入响应，并通过 `WeakMap` 缓存每条日志的标准化搜索字段、域名和资源类型。
6. 初始 HTTP 历史与实时 WebSocket 日志按 ID 合并，避免加载竞态覆盖新日志。
7. 请求详情仍只保留最近 200 条，未扩大 body 内存占用。

- CPU：Apple M4
- 内存：16 GiB

- 每种吞吐场景运行 7 次，取中位数

## 前后性能数据

工作负载：已有容量达到上限，连续接收 500 条日志，并执行当前过滤和时间排序派生路径。

| 场景 | 容量 | 批大小 | 中位耗时 |
|---|---:|---:|---:|
| 修改前实际实现 | 1,000 | 1 | 181.93ms |
| 仅提高上限，未优化 | 10,000 | 1 | 1,704.29ms |
| 修改后实现 | 10,000 | 25 | **7.36ms** |

修改代码前独立采集到的旧实现中位数为 168.26ms（1k）和 1,703.47ms（10k）。与最终基准中的旧算法复测结果接近，10k 结果偏差小于 0.1%。
